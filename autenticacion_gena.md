*Reutilizable en cualquier proyecto Oracle APEX — Creative90*

## 1. Estructura de tablas

```sql
CREATE TABLE gena_aplicacion (
    id_aplicacion     NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    nombre            VARCHAR2(150) NOT NULL,
    tipo_aplicacion   VARCHAR2(30),   -- 'APEX', 'FORMS', 'MOVIL'
    codigo_referencia VARCHAR2(100),  -- app_id de APEX
    fecha_creacion    DATE DEFAULT SYSDATE NOT NULL
);

CREATE TABLE gena_pagina (
    id_pagina         NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    id_aplicacion     NUMBER NOT NULL,
    nombre            VARCHAR2(150) NOT NULL,
    codigo_referencia VARCHAR2(100),
    fecha_creacion    DATE DEFAULT SYSDATE NOT NULL,
    CONSTRAINT fk_pagina_aplicacion
        FOREIGN KEY (id_aplicacion) REFERENCES gena_aplicacion(id_aplicacion)
);

CREATE TABLE gena_usuario (
    id_usuario       NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    id_aplicacion    NUMBER NOT NULL,
    id_candidato     NUMBER,          -- nullable: solo apps con candidatos
    nombre           VARCHAR2(150) NOT NULL,
    login            VARCHAR2(50)  NOT NULL,
    password_hash    VARCHAR2(256) NOT NULL,
    activo           CHAR(1) DEFAULT 'S' NOT NULL CHECK (activo IN ('S','N')),
    cambio_requerido CHAR(1) DEFAULT 'S' NOT NULL CHECK (cambio_requerido IN ('S','N')),
    fecha_creacion   DATE DEFAULT SYSDATE NOT NULL,
    CONSTRAINT uq_usuario_login UNIQUE (login),
    CONSTRAINT fk_usuario_aplicacion
        FOREIGN KEY (id_aplicacion) REFERENCES gena_aplicacion(id_aplicacion)
);

CREATE TABLE gena_rol (
    id_rol        NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    id_aplicacion NUMBER NOT NULL,
    descripcion   VARCHAR2(100) NOT NULL,
    CONSTRAINT fk_rol_aplicacion
        FOREIGN KEY (id_aplicacion) REFERENCES gena_aplicacion(id_aplicacion)
);

CREATE TABLE gena_usuario_rol (
    id_usuario_rol NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    id_usuario     NUMBER NOT NULL,
    id_rol         NUMBER NOT NULL,
    CONSTRAINT fk_usuariorol_usuario
        FOREIGN KEY (id_usuario) REFERENCES gena_usuario(id_usuario),
    CONSTRAINT fk_usuariorol_rol
        FOREIGN KEY (id_rol) REFERENCES gena_rol(id_rol),
    CONSTRAINT uq_usuario_rol UNIQUE (id_usuario, id_rol)
);

CREATE TABLE gena_rol_pagina (
    id_rol_pagina NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    id_rol        NUMBER NOT NULL,
    id_pagina     NUMBER NOT NULL,
    CONSTRAINT fk_rolpagina_rol
        FOREIGN KEY (id_rol) REFERENCES gena_rol(id_rol),
    CONSTRAINT fk_rolpagina_pagina
        FOREIGN KEY (id_pagina) REFERENCES gena_pagina(id_pagina),
    CONSTRAINT uq_rol_pagina UNIQUE (id_rol, id_pagina)
);

-- Índices
CREATE INDEX idx_gena_pagina_aplicacion  ON gena_pagina(id_aplicacion);
CREATE INDEX idx_gena_usuario_aplicacion ON gena_usuario(id_aplicacion);
CREATE INDEX idx_gena_rol_aplicacion     ON gena_rol(id_aplicacion);
CREATE INDEX idx_gena_usuariorol_usuario ON gena_usuario_rol(id_usuario);
CREATE INDEX idx_gena_usuariorol_rol     ON gena_usuario_rol(id_rol);
CREATE INDEX idx_gena_rolpagina_rol      ON gena_rol_pagina(id_rol);
CREATE INDEX idx_gena_rolpagina_pagina   ON gena_rol_pagina(id_pagina);
```

## 2. Decisiones de diseño

- **Un usuario pertenece a una sola aplicación** — `id_aplicacion` NOT NULL con FK.
- **LOGIN es UNIQUE global** — aislamiento por app/candidato a nivel de negocio.
- **`id_candidato`** en `gena_usuario` — nullable, solo aplica en apps con módulo multi-candidato.
- **Roles por aplicación** — `gena_rol.id_aplicacion` evita mezcla entre proyectos.
- **`cambio_requerido`** DEFAULT `'S'` — al crear usuario queda pendiente; se pone `'N'` al cambiar contraseña.
- **Contraseña inicial = UPPER(login)** — convención Creative90. El usuario debe cambiarla al primer ingreso.
- **Contraseña con `ORA_HASH`** — llamar siempre dentro de `SELECT ... FROM DUAL` en PL/SQL.
- **`v('APP_ID')` NO funciona en contexto de autenticación** — retorna NULL. No usarlo en `fu_autenticar`.
- **`UPPER(login)`** en ambos lados del WHERE — APEX convierte el username a mayúsculas al autenticar.
- **`id_aplicacion` en `gena_aplicacion` es secuencia propia** — NO coincide necesariamente con el APP_ID de APEX. El APP_ID de APEX se guarda en `codigo_referencia`.
- **`fu_autenticar` filtra por `codigo_referencia`** — hace JOIN con `gena_aplicacion` usando `v('APP_ID')` para aislar usuarios por app. Sin este filtro, usuarios de una app pueden entrar a otra.

## 3. Función de autenticación

```sql
CREATE OR REPLACE FUNCTION fu_autenticar(
    p_username IN VARCHAR2,
    p_password IN VARCHAR2
) RETURN BOOLEAN IS
    v_count NUMBER;
BEGIN
    SELECT COUNT(*)
    INTO   v_count
    FROM   gena_usuario   u,
           gena_aplicacion a
    WHERE  u.id_aplicacion     = a.id_aplicacion
    AND    UPPER(u.login)      = UPPER(p_username)
    AND    u.password_hash     = TO_CHAR(ORA_HASH(p_password))
    AND    u.activo            = 'S'
    AND    a.codigo_referencia = TO_CHAR(v('APP_ID'));

    RETURN v_count > 0;
EXCEPTION
    WHEN OTHERS THEN
        RETURN FALSE;
END;
```

> **Importante:** `v('APP_ID')` SÍ funciona correctamente en el contexto del Authentication Scheme (no en el proceso interno de autenticación). El filtro por `codigo_referencia` garantiza que cada usuario solo puede ingresar a su app correspondiente.

## 4. Configuración Authentication Scheme en APEX

- **Shared Components → Authentication Schemes → Create**
- **Name**: `AUTH_GENA`
- **Scheme Type**: `Custom`
- **Authentication Function Name**: `fu_autenticar`
- Marcar como **Current**

> La función `fu_autenticar` es compartida en la base de datos — todas las apps del grupo usan la misma función sin necesidad de crearla nuevamente.

## 5. Authorization Schemes

### ES_ADMIN
- **Evaluation Point: Once per page view** — NO usar "Once per session" (cachea FALSE para usuarios admin en sesiones distintas).

```sql
DECLARE
    v_count NUMBER;
BEGIN
    SELECT COUNT(*)
    INTO   v_count
    FROM   gena_usuario_rol ur,
           gena_rol r,
           gena_usuario u
    WHERE  ur.id_rol    = r.id_rol
    AND    ur.id_usuario = u.id_usuario
    AND    UPPER(u.login) = UPPER(:APP_USER)
    AND    r.descripcion  = 'Administrador';

    RETURN v_count > 0;
EXCEPTION
    WHEN OTHERS THEN RETURN FALSE;
END;
```

### ES_USUARIO_ACTIVO
```sql
DECLARE
    v_count NUMBER;
BEGIN
    SELECT COUNT(*)
    INTO   v_count
    FROM   gena_usuario
    WHERE  UPPER(login) = UPPER(:APP_USER)
    AND    activo = 'S';

    RETURN v_count > 0;
EXCEPTION
    WHEN OTHERS THEN RETURN FALSE;
END;
```

> **Importante:** Para Server-side Condition de regiones restringidas a ADMIN, usar Type **"Rows returned"** con SQL directo a `gena_usuario_rol` — no usar el Authorization Scheme `ES_ADMIN` como condición de región (puede cachear mal entre usuarios).

```sql
SELECT 1
FROM   gena_usuario_rol ur,
       gena_rol r,
       gena_usuario u
WHERE  ur.id_rol     = r.id_rol
AND    ur.id_usuario = u.id_usuario
AND    UPPER(u.login) = UPPER(:APP_USER)
AND    r.descripcion  = 'Administrador'
```

## 6. Scripts reutilizables

### Registrar nueva aplicación

```sql
-- IMPORTANTE: id_aplicacion es secuencia propia, NO usar el APP_ID de APEX.
-- El APP_ID de APEX va en codigo_referencia.
INSERT INTO gena_aplicacion (nombre, tipo_aplicacion, codigo_referencia)
VALUES ('NOMBRE_APP', 'APEX', 'APP_ID_APEX');
COMMIT;
```

### Crear usuario administrador inicial

```sql
-- Convención: primer admin = ADMIN1, siguiente app = ADMIN2, etc.
INSERT INTO gena_usuario (id_aplicacion, id_candidato, nombre, login, password_hash, activo, cambio_requerido)
VALUES (
    (SELECT id_aplicacion FROM gena_aplicacion WHERE codigo_referencia = 'APP_ID_APEX'),
    NULL,
    'Administrador NombreApp',
    'ADMIN1',
    TO_CHAR(ORA_HASH('ADMIN1')),
    'S',
    'N'   -- 'N' hasta que esté lista la página 9998
);
COMMIT;
```

### Crear rol y asignarlo al admin

```sql
DECLARE
    v_id_rol     NUMBER;
    v_id_usuario NUMBER;
    v_id_app     NUMBER;
BEGIN
    SELECT id_aplicacion INTO v_id_app
    FROM   gena_aplicacion
    WHERE  codigo_referencia = 'APP_ID_APEX';

    INSERT INTO gena_rol (id_aplicacion, descripcion)
    VALUES (v_id_app, 'Administrador')
    RETURNING id_rol INTO v_id_rol;

    SELECT id_usuario INTO v_id_usuario
    FROM   gena_usuario
    WHERE  login = 'ADMIN1'
    AND    id_aplicacion = v_id_app;

    INSERT INTO gena_usuario_rol (id_usuario, id_rol)
    VALUES (v_id_usuario, v_id_rol);

    COMMIT;
    DBMS_OUTPUT.PUT_LINE('OK - Rol id: ' || v_id_rol || ' asignado a ADMIN1');
EXCEPTION
    WHEN OTHERS THEN
        ROLLBACK;
        RAISE_APPLICATION_ERROR(-20001, 'ERROR: ' || SQLERRM);
END;
```

### Nuevo candidato + usuario administrador (apps con candidatos)

```sql
DECLARE
    v_nombre_candidato   VARCHAR2(150) := 'NOMBRE';
    v_login              VARCHAR2(50)  := 'LOGIN';
    v_cargo_candidato    VARCHAR2(100) := 'CARGO';
    v_partido            VARCHAR2(100) := 'PARTIDO';
    v_eslogan            VARCHAR2(500) := 'ESLOGAN';
    v_sitio_web          VARCHAR2(300) := 'https://';
    v_foto_url           VARCHAR2(500) := 'https://raw.githubusercontent.com/sistemascreative90/public/main/ARCHIVO.jpg';
    v_nombre_cabecera    VARCHAR2(150) := 'NOMBRE_CABECERA';
    v_apellido_cabecera  VARCHAR2(150) := 'APELLIDO_CABECERA';
    v_cargo_cabecera     VARCHAR2(100) := 'CARGO_CABECERA';

    v_id_candidato       NUMBER;
    v_id_usuario         NUMBER;
BEGIN
    INSERT INTO campanha_candidato (
        nombre, cargo, partido, eslogan, sitio_web, foto_url,
        nombre_cabecera, apellido_cabecera, cargo_cabecera
    ) VALUES (
        v_nombre_candidato, v_cargo_candidato, v_partido, v_eslogan, v_sitio_web, v_foto_url,
        v_nombre_cabecera, v_apellido_cabecera, v_cargo_cabecera
    ) RETURNING id_candidato INTO v_id_candidato;

    INSERT INTO gena_usuario (
        id_aplicacion, id_candidato, nombre, login, password_hash, activo, cambio_requerido
    ) VALUES (
        1, v_id_candidato, v_nombre_candidato,
        UPPER(v_login),
        TO_CHAR(ORA_HASH(UPPER(v_login))),
        'S', 'S'
    ) RETURNING id_usuario INTO v_id_usuario;

    INSERT INTO gena_usuario_rol (id_usuario, id_rol)
    SELECT v_id_usuario, id_rol FROM gena_rol
    WHERE  id_aplicacion = 1
    AND    descripcion IN ('Administrador', 'Visitador');

    COMMIT;

    DBMS_OUTPUT.PUT_LINE('OK - Candidato id: ' || v_id_candidato || ' | Login: ' || UPPER(v_login));
EXCEPTION
    WHEN OTHERS THEN
        ROLLBACK;
        RAISE_APPLICATION_ERROR(-20001, 'ERROR: ' || SQLERRM);
END;
```

## 7. Página de cambio de contraseña (Page 9998)

Reservada en todas las apps Creative90. Se copia desde app 101 via Export/Import de página.

**Procedimiento de copia entre apps de distintos workspaces:**
1. App origen → Page Designer → página 9998 → Export Page → descargar `.sql`
2. Editar el `.sql`: cambiar `p_default_application_id` al ID de la app destino
3. Cambiar `p_default_id_offset` de `0` a `20000000000000000` (evita colisión de IDs)
4. App destino → SQL Workshop → SQL Scripts → Upload → ejecutar

**Región:** `RGN_FRM_CAMBIO_CLAVE`

**Ítems:** `P9998_CLAVE_ACTUAL`, `P9998_CLAVE_NUEVA`, `P9998_CLAVE_CONFIRMA` (todos Password)

**Botón:** `BTN_PG_GUARDAR` — Hot, Submit

**Validaciones:**
- `VAL_CLAVE_ACTUAL_REQ` — Item is NOT NULL
- `VAL_CLAVE_NUEVA_REQ` — Item is NOT NULL
- `VAL_CLAVES_COINCIDEN` — PL/SQL Expression: `:P9998_CLAVE_NUEVA = :P9998_CLAVE_CONFIRMA`

**Proceso `PRC_AS_PRO_CAMBIO_CLAVE` — After Submit:**
```sql
DECLARE
    v_hash_actual VARCHAR2(256);
    v_hash_nuevo  VARCHAR2(256);
BEGIN
    SELECT TO_CHAR(ORA_HASH(:P9998_CLAVE_ACTUAL)) INTO v_hash_actual FROM DUAL;
    SELECT TO_CHAR(ORA_HASH(:P9998_CLAVE_NUEVA))  INTO v_hash_nuevo  FROM DUAL;

    UPDATE gena_usuario
    SET    password_hash    = v_hash_nuevo,
           cambio_requerido = 'N'
    WHERE  UPPER(login)     = UPPER(:APP_USER)
    AND    password_hash    = v_hash_actual;

    IF SQL%ROWCOUNT = 0 THEN
        RAISE_APPLICATION_ERROR(-20001, 'La contraseña actual es incorrecta');
    END IF;

    COMMIT;

    apex_util.redirect_url(
        apex_page.get_url(p_page => 9999, p_request => 'LOGOUT')
    );
END;
```

**Branch:** `BRH_OK_TO_P1` — After Processing, When Button Pressed = `BTN_PG_GUARDAR`, redirige a página 1.

## 8. Application Process — Controlador de cambio obligatorio

**`PRC_BH_CHK_CAMBIO_CLAVE` — Point: On Load: Before Header (global)**

Redirige automáticamente a página 9998 si `cambio_requerido = 'S'`. Excluir páginas 9998 y 9999 para evitar loop infinito.

**Versión para apps sin candidatos (ej. AGRPRO):**
```sql
DECLARE
    v_cambio VARCHAR2(1);
BEGIN
    SELECT cambio_requerido
    INTO   v_cambio
    FROM   gena_usuario
    WHERE  UPPER(login)  = UPPER(:APP_USER)
    AND    id_aplicacion = (
               SELECT id_aplicacion FROM gena_aplicacion
               WHERE  codigo_referencia = TO_CHAR(:APP_ID)
           );

    IF v_cambio = 'S' AND :APP_PAGE_ID NOT IN ('9998', '9999') THEN
        apex_util.redirect_url(
            apex_page.get_url(p_page => 9998)
        );
    END IF;
EXCEPTION
    WHEN NO_DATA_FOUND THEN NULL;
END;
```

**Versión para apps con candidatos (ej. CAMPANHA):**
```sql
DECLARE
    v_cambio          CHAR(1);
    v_id_candidato    NUMBER;
    v_nombre          VARCHAR2(150);
    v_cargo           VARCHAR2(100);
    v_eslogan         VARCHAR2(500);
    v_foto            VARCHAR2(500);
    v_nombre_cab      VARCHAR2(150);
    v_apellido_cab    VARCHAR2(150);
    v_cargo_cab       VARCHAR2(100);
BEGIN
    SELECT u.cambio_requerido, u.id_candidato,
           c.nombre, c.cargo, c.eslogan, c.foto_url,
           c.nombre_cabecera, c.apellido_cabecera, c.cargo_cabecera
    INTO   v_cambio, v_id_candidato,
           v_nombre, v_cargo, v_eslogan, v_foto,
           v_nombre_cab, v_apellido_cab, v_cargo_cab
    FROM   gena_usuario u,
           campanha_candidato c
    WHERE  u.id_candidato  = c.id_candidato
    AND    UPPER(u.login)  = UPPER(:APP_USER);

    :AI_ID_CANDIDATO      := v_id_candidato;
    :AI_NOMBRE_CANDIDATO  := v_nombre;
    :AI_CARGO_CANDIDATO   := v_cargo;
    :AI_ESLOGAN           := v_eslogan;
    :AI_FOTO_CANDIDATO    := v_foto;
    :AI_NOMBRE_CABECERA   := v_nombre_cab;
    :AI_APELLIDO_CABECERA := v_apellido_cab;
    :AI_CARGO_CABECERA    := v_cargo_cab;

    IF :APP_PAGE_ID NOT IN ('9998', '9999') AND v_cambio = 'S' THEN
        apex_util.redirect_url(apex_page.get_url(p_page => 9998));
    END IF;
EXCEPTION
    WHEN NO_DATA_FOUND THEN NULL;
END;
```

## 11. Convenciones Creative90

- Usuario admin inicial por app: `ADMIN1`, `ADMIN2`, `ADMIN3`... según orden de creación
- Contraseña inicial = UPPER(login)
- `cambio_requerido = 'N'` al crear el admin inicial si la página 9998 aún no existe
- `cambio_requerido = 'S'` para todos los demás usuarios nuevos

## 12. Consideraciones importantes

- `ORA_HASH` en PL/SQL debe llamarse dentro de `SELECT ... FROM DUAL` — no funciona como función directa.
- `v('APP_ID')` retorna NULL en el proceso interno de autenticación — no usarlo dentro de `fu_autenticar` para filtrar. El filtro se hace via `codigo_referencia` que sí es accesible desde el Authentication Scheme.
- `ES_ADMIN` debe ser **Once per page view** — Once per session cachea el resultado y falla con múltiples usuarios admin.
- Server-side Condition de regiones ADMIN: usar **"Rows returned"** con SQL directo, no Authorization Scheme.
- `id_aplicacion` en `gena_aplicacion` es una secuencia propia — NO forzar el APP_ID de APEX como `id_aplicacion`. Siempre dejar que la secuencia genere el valor y usar `codigo_referencia` para el APP_ID de APEX.
- Al modificar `id_aplicacion` en `gena_aplicacion` hay que deshabilitar las FKs hijas primero: `FK_PAGINA_APLICACION`, `FK_USUARIO_APLICACION`, `FK_ROL_APLICACION`.
- Contraseña inicial = UPPER(login) — convención Creative90 para todos los usuarios nuevos.
